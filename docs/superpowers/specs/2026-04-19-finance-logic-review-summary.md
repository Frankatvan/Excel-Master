# 成本归集逻辑审查与转型战略总结 (2026-04-19)

## 1. 逻辑审查结论 (Classification Engine Review)
当前 `check_finance.py` 中的分类引擎高度还原了设计文档的决策树，逻辑分层清晰。

### 核心发现与风险点：
- **ACC/RACC 配对金额风险 (高)**：当前 `_make_final_detail_pair_key` 使用原始金额作为匹配键。若财务记录中预提为正、冲回为负，逻辑将因正负号不同导致配对失败，进而使 `RACC` 误分类为 `EXP`。
- **Cost Code 提取脆弱性 (中)**：`_extract_tail_int` 严格截取末尾 3 位。若 Cost Code 为数值型（如 50）或短字符串（"50"），将返回 `None`，导致分类退化为 `Direct` 或 `EXP`。
- **结算日期 Fallback 逻辑**：在 `C/O Date` 缺失时回溯读取表内现有日期，这可能保留不符合新规则的旧数据，与“缺失则不判定为已结算”的设计初衷略有偏差。
- **业务规则确认**：确认 `Payable` 表记录均为实付（Actuals），**不应包含 ACC（预提）分类**。

## 2. 云端化与多项目转型评估
### 现状分析：
- **解耦程度**：核心分类函数（`_classify_xxx`）已具备良好的原子性，逻辑与 IO 分离度约 60%。
- **数据源获取**：WBS 系统数据（Unit budget, Payable, Final Detail, Draw request report）未来将由 Excel 导出改为 API 直接获取。

## 3. 战略共识
- **不打断、不提前重构**：当前继续以 Google Sheets 作为“逻辑实验室”是正确路径。
- **演进式路线**：
    1.  **第一阶段 (当前)**：基于 Excel 导出件，锁死业务逻辑与算法。
    2.  **第二阶段 (上云前)**：在本地脚本中接入 WBS API。解决鉴权、数据清洗与字段对齐问题，保持 Sheets 为输出，确保“ API 数据输入”下的逻辑一致性。
    3.  **第三阶段 (全流程上云)**：将成熟的 API 接入模块与业务计算逻辑整体迁移至云端架构（DB + Dashboard）。

## 4. 下一步关注
- 在后续需求收集时，重点确认财务数据中金额正负号的对冲习惯。
- 保持核心逻辑函数的“纯净性”，确保其输入输出与特定的数据载体（Sheets 或数据库）解耦。
