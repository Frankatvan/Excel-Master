# 项目注册表与动态 109 Sheet 设计

## 背景

当前审计工作台把项目展示和 109 相关分析建立在两个默认前提上：

1. 工作台里的项目来自 Supabase `projects` 注册表，而不是“用户手里有某个 Google Sheet 链接”
2. 系统默认每个项目的 109 工作表都固定叫 `109`

这两个前提已经不再满足新的业务要求：

- 历史项目 `Project Ledger_WBWT Sandy Cove_4.17.2026` 需要直接出现在工作台里，而不是重新新建
- 新项目需要输入三位数字序列号，并把该序列号作为项目自己的 109 sheet 标题
- 系统以后不能再硬编码 `"109"`，而必须按项目自己的三位序列号 sheet 读写
- Google Sheet 文件名需要在“创建时”和“提交审计确认锁定时”按不同日期规则重命名
- 按钮 `109表初验通过` 需要改名为 `提交审计确认`

因此，本轮设计要同时解决“历史项目接入”和“109 sheet 动态化”两类问题，并保证后续所有项目都遵循统一规则。

## 目标

本轮设计的目标是：

- 让历史项目 `WBWT Sandy Cove` 作为正式项目回填到工作台项目注册表
- 为每个项目建立唯一的三位数字序列号，并把它作为 109 sheet 标题的唯一真源
- 让所有原来依赖 `"109"` 的逻辑改为依赖 `projects.sheet_109_title`
- 让 Google Sheet 文件名在创建和锁定两个时间点按统一命名规范自动重命名
- 保持现有项目流程、日志体系、保护规则和前端工作台结构不被打散

## 非目标

本轮不做以下事情：

- 不做“运行时自动猜测哪个三位数字 sheet 是 109 sheet”
- 不把项目列表从 Supabase 注册表改成纯 Google Drive 扫描
- 不重置历史项目的数据、状态、保护规则或人工录入内容
- 不改变当前阶段日志、锁定机制、编辑追踪机制的业务含义
- 不重构 109 计算公式本身，只改其 sheet 定位方式

## 方案选择

### 方案 A：在项目注册表中显式保存 `sheet_109_title`

为 `projects` 表新增一个字段 `sheet_109_title`，值为项目唯一的三位数字序列号，例如 `109`、`237`。  
系统中所有原来硬编码 `"109"` 的逻辑，统一改成：

1. 先按当前项目读取 `sheet_109_title`
2. 再使用该标题拼接 Sheet range、保护规则和分析读取逻辑

这是本次采用的方案。

### 方案 B：运行时自动寻找三位数字 sheet

每次执行时直接扫描工作簿，自动寻找唯一的三位数字工作表并把它当成 109。

该方案不采用，原因是：

- 只要工作簿里后续出现别的三位数字 sheet，就可能误判
- 审计链路里缺少稳定锚点，排障成本高
- 历史项目接入、新项目创建、前端分析会分别实现不同的推断逻辑，容易漂移

## 数据模型设计

### `projects` 表新增字段

在 Supabase `projects` 表中新增：

- `sheet_109_title TEXT NOT NULL`

字段约束：

- 必须是严格三位数字字符串
- 在全系统内唯一
- 一旦项目创建成功，该字段默认不可为空

推荐增加唯一约束或唯一索引：

- `UNIQUE(sheet_109_title)`

### 历史项目回填

将现有历史项目 `Project Ledger_WBWT Sandy Cove_4.17.2026` 回填为正式项目记录：

- `name = WBWT Sandy Cove`
- `spreadsheet_id = 1g17SLJDKIT9rM5Vi4xQoD_pa4byvDPE6gJvRwi5bQDw`
- `sheet_109_title = 109`
- `owner_email = 当前项目 Owner`
- `user_id_sub = 当前登录用户对应的 sub`

兼容原则：

- 不重置历史项目状态
- 不修改历史项目现有数据
- 不强制重命名其 109 sheet，因为它当前已经与目标序列号 `109` 一致

## 新项目创建设计

### 前端输入项

“添加新项目”弹窗改为三个输入项：

- `Project Name`
- `Project Owner`
- `三位序列号`

### 三位序列号规则

输入校验规则：

- 必须是严格三位数字
- 允许范围为 `000` 到 `999`
- 全系统唯一，不能和已有项目重复
- 前端即时校验
- 后端创建接口二次校验

错误提示应明确区分：

- 格式错误：不是三位数字
- 冲突错误：该序列号已被其他项目占用

### 创建流程

创建成功时，系统按以下顺序执行：

1. 复制当前模板工作簿
2. 将模板中的 `109` sheet 重命名为用户输入的三位数字序列号
3. 将 109 表头中的项目名称和项目负责人写入
4. 清空外部数据区与浅灰人工录入区
5. 初始化项目状态表与项目日志表
6. 在 Supabase `projects` 表中写入项目记录，包括 `sheet_109_title`

### 回滚原则

若任一步骤失败：

- 已复制出的 Google Sheet 必须清理
- Supabase 项目记录不得保留半成品
- 前端返回明确错误

## Google Sheet 文件命名规则

### 创建时文件名

项目创建完成后，Google Sheet 文件名改为：

`Project Ledger_<三位序列号>_<Project Name>_<创建日期>`

示例：

`Project Ledger_109_WBWT Sandy Cove_4.24.2026`

### 锁定后文件名

当用户点击 `提交审计确认` 后，若动作成功：

- 记录锁定日志
- 锁定数据
- 将 Google Sheet 文件名改为：

`Project Ledger_<三位序列号>_<Project Name>_<锁定日期>`

示例：

`Project Ledger_109_WBWT Sandy Cove_5.2.2026`

### 日期格式

日期格式固定为：

- `M.D.YYYY`
- 不补零

### 命名边界

本轮只控制 Google Sheet 文件名，不改变：

- 项目在前端工作台中的显示标题
- `Project Name` 字段本身
- 其他工作表名称，除项目 109 sheet 外

## 109 动态 Sheet 标题设计

### 真源定义

系统不再以字符串 `"109"` 为默认值，而是以：

- `projects.sheet_109_title`

作为项目 109 sheet 的唯一真源。

所有需要访问 109 数据的代码都应先解析当前项目的 `sheet_109_title`，再拼接工作表引用。

### 影响范围

#### 1. Worker / 后端计算层

所有原来写死 `"109"` 的范围引用与 Sheet 元数据定位都改为动态标题，包括但不限于：

- 公式同步
- 109 布局控制
- 109 手工录入区保护
- 109 live status 探测
- 109 对比读取
- 任何 `109!A:ZZ`、`'109'!C5`、`'109'!A1:R80` 等引用

#### 2. 前端分析层

前端依赖 109 的展示逻辑统一改为动态标题，包括：

- 首页 KPI 中依赖 109 的摘要
- `109 对比` 页签
- 同步数据后读取的 109 分析结果
- live sheet status 中对 109 保护状态的展示

#### 3. 状态动作层

与 109 绑定的动作文案与结果更新同步调整：

- `109表初验通过` 改名为 `提交审计确认`
- 其动作语义仍然表示：用户确认 109 初验通过并进入锁定阶段
- 成功后触发锁定日志与文件名重命名

## 历史兼容与迁移策略

### 历史项目

历史项目 `WBWT Sandy Cove` 的迁移策略为：

- 先补项目注册记录
- `sheet_109_title = 109`
- 不动现有 sheet 标题
- 不动现有工作簿数据

### 新项目模板

模板中仍可以保留初始 sheet 名 `109`，但新项目创建时必须在 bootstrap 阶段立即重命名为用户输入的三位数字。

### 禁止自动推断

无论历史项目还是新项目，都不依赖运行时自动推断数字 sheet：

- 历史项目依赖回填后的 `sheet_109_title`
- 新项目依赖创建时写入的 `sheet_109_title`

## 错误处理

### 创建项目

若出现以下错误，需要给出明确报错：

- 三位序列号格式错误
- 三位序列号重复
- 模板复制失败
- 109 sheet 重命名失败
- 项目注册失败
- 文件命名失败

### 审计确认

点击 `提交审计确认` 时，若锁定成功但文件重命名失败，系统必须：

- 保留锁定状态和锁定日志
- 记录一条失败日志说明“文件名更新失败”
- 前端提示用户该项目已锁定，但文件名未完成更新

这是因为“锁定业务状态”比“文件命名成功”更重要，不能因为命名失败回滚锁定。

## 测试策略

### 后端 / API

需要覆盖：

- 历史项目回填记录可被项目列表接口识别
- 新项目创建时三位序列号格式校验
- 新项目创建时三位序列号唯一性校验
- 项目创建后 `sheet_109_title` 正确写入
- 模板复制后 109 sheet 被成功重命名
- 项目锁定后文件名按锁定日期更新

### Worker / Sheet 逻辑

需要覆盖：

- 动态 109 sheet 标题下的公式同步
- 动态 109 sheet 标题下的保护规则同步
- 动态 109 sheet 标题下的 live status 读取

### 前端

需要覆盖：

- 添加新项目弹窗新增序列号输入项
- 序列号格式错误时的提示
- 序列号冲突时的提示
- 按钮文案从 `109表初验通过` 改为 `提交审计确认`
- 历史项目出现在项目列表中

## 验收标准

完成后应满足以下结果：

1. 登录工作台时，历史项目 `WBWT Sandy Cove` 能作为正式项目显示
2. 历史项目的 109 相关分析、保护状态、同步逻辑正常工作
3. 新建项目时必须填写三位数字序列号，且不能重复
4. 新建项目后，模板中的 `109` sheet 会被立即改名为该三位数字
5. 系统不再依赖硬编码 `"109"` 访问 109 sheet
6. `109表初验通过` 按钮统一改名为 `提交审计确认`
7. 创建成功后文件名按创建日期命名
8. 提交审计确认成功后文件名按锁定日期重命名

## 风险与注意事项

### 风险 1：硬编码 `"109"` 引用遗漏

该风险最高。  
处理方式是对前后端、worker、live status、audit dashboard 做全文搜索并逐个替换，不允许只改创建链路。

### 风险 2：历史项目回填不完整

若只回填 `spreadsheet_id` 但不回填 `sheet_109_title`，则运行时仍会无法定位 109 sheet。

### 风险 3：文件重命名与锁定动作顺序

锁定动作不能依赖文件重命名成功。  
重命名失败时，只能记录失败日志并提示，不应撤销锁定。

### 风险 4：三位序列号唯一性约束只做前端校验

必须由数据库或后端接口做最终唯一性控制，否则并发创建时会冲突。
