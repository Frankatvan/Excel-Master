# AiWB 项目最高使命与架构宪章 (Mission Control)

## 1. 项目本质 (The Core)
AiWB 是一个**符合德勤审计标准（HKFRS 15）**的项目级 GMP 财务重分类与核算引擎。
- **业务目的**：通过对 WBS 原始成本数据的重分类（ROE/ACC/RACC），核算 GMP 合同的真实收入、成本与毛利。
- **核心逻辑**：基于“投入法（Cost-to-Cost Input Method）”的 5 步核算模型。
- **交付目标**：指导会计核算，并出具获得四大认可的审计报告。

## 2. 核心架构准则 (Architecture Principles)

### A. 规则锚定 (Rule Anchoring)
- **准则**：逻辑规则由德勤审计确认后即进入“冻结状态”，不随日常操作变动。
- **影响**：Rules ID 系统和重分类判断代码必须具备高内聚性，非审计变更不得随意改动。

### B. 白盒审计 (White-box Audit)
- **准则**：除了极复杂的 Rule ID 判断逻辑外，109 表的所有计算逻辑必须以 **Excel 公式** 形式展现。
- **目的**：确保审计师点开单元格即可复核，符合“可追溯、可解释”的审计要求。

### C. 数据安全与回滚 (Safety & Rollback)
- **保护**：系统需具备锁定功能（Protected Ranges），防止非技术同事误改公式区。
- **回滚**：必须具备版本快照功能，当操作失误或逻辑崩溃时，可一键恢复至上一个干净版本。

### D. 语义解耦 (Semantic Mapping)
- **映射**：逻辑不再依赖硬编码行号。通过 `Scoping` 表中的科目对应关系（1~6 类）进行动态归集。

## 3. 标准化工作流 (Standard Workflow)

1. **数据准入**：自动/手动从 WBS 导入 Payable、Unit Budget、Final Detail 等原始数据。
2. **科目定义**：在 `Scoping` 表中完成 GMP 标识（E列=1）及科目类别映射。
3. **一键核算 (The Big Button)**：
   - 执行重分类：为 Payable/Final Detail 每一行打上 Rule ID。
   - 驱动汇总：109 表根据重分类结果自动生成公式并计算。
4. **自动化稽核 (Audit)**：
   - 识别异常（如对账差异、逻辑冲突），并将其推送到前端 UI 进行提示。
5. **对账与出账**：
   - 借助 AI 辅助完成乱序的甲方对账。
   - 输出**会计分录调整表**，直接指导财务做账。

## 4. 扩展性与数据聚合 (Scalability & Aggregation)
- **多项目管理**：支持 100+ 项目，一个项目一个 Google Sheet。
- **本地数据库 (Meta DB)**：在本地建立数据库，实时同步各项目的核心财务指标（POC, Margin, Variances）。
- **动态看板 (Streamlit Dashboard)**：在 Streamlit 界面提供全局视图，支持动态查询、对比 100+ 项目的实时状况，而非依赖逐个打开 Google Sheets。
- **项目内视图**：在每个项目的 Google Sheet 中单独增加一个 `Summary` 工作表，展示核心核算结果及审计摘要。
- **云端预留**：架构需支持未来迁移至 AWS，实现多租户、多版本、每月连续跟踪。

## 5. 工作模式与信任契约 (Working Mode & Trust Pact)
1. **架构师-Codex 自动链路（手脑合一）**：采用Gemini CLI是架构师（大脑），Codex是执行者，代码编写和修改，Gemini CLI最否负责验收的模式，代码尽可能调用Codex Agent去修改，决策即执行，直接操作文件系统完成代码和文件的修改、创建与重构，严禁手动粘贴代码。
2. **沉默授权 (No Micromanagement)**：任务开始前需要经过我讨论确认，任务一旦确认禁止询问细碎确认，AI 拥有 100% 的代码控制权与重构权，只需对业务目标和审计结果负责。
3. **长程任务模式 (Long-Burst Execution)**：全量突击，一次性完成“文档审计 -> 架构规划 -> 代码重写 -> 测试验证”完整链条。仅在逻辑完全自相矛盾、API 瘫痪或数据损坏风险时才允许中断反馈。
4. **TDD 闭环验证**：所有修改必须配合测试验证，任务结束后交付简明报告（核心模块修改、测试结果、状态更新）。
